System Awareness Investigation

Why I lose track of what's already shipped, and how to fix it before the system breaks at scale. May 16, 2026.

TL;DR. Two real problems surfaced today. First, I treated old lastUpdated timestamps as the canonical "what's stale" signal, but that field gets bumped every time I touch a piece, so newly-shipped content and recently-refreshed legacy content look identical. The fix is small: publication-date-aware audit logic + a "what I touched this session" memory hook. Second, the Notion brain is on track to exceed context limits within ~3 months as Activity Log + Daily Snapshots + Patterns Registry compound. The fix is bigger: a digest layer that auto-summarizes high-volume databases nightly, plus per-skill context scoping so /seo-brain doesn't fetch everything for every task. Five concrete proposals below, ordered by leverage.

2. Root causes — three structural gaps

Gap 1: The skills don't have session memory

/seo-brain, /draft-content, /ship-content, /critique-content each load fresh context every invocation. None of them know that the previous skill in the loop wrote anything. The Activity Log database in Notion is the cross-session audit trail, but the skills don't read it before acting — they only write to it after acting.

Consequence: I can ship a piece in batch 11, propose it again as "stale" in batch 13, and the system has no immune response. The Activity Log knows; the skill doesn't.

Gap 2: lastUpdated is overloaded

The field means "any change to this doc" — including title patches, content refreshes, and new publications. It's the obvious field to query, but it conflates three different events:

  • "This is a new piece" (used to mean: lastUpdated == publishedDate)
  • "This was refreshed" (publishedDate old, lastUpdated recent)
  • "This was patched for a small fix" (both dates ancient, lastUpdated bumped by a single field edit)

Sanity doesn't distinguish these. Our schema doesn't either. So every audit needs to manually parse the relationship between publishedDate, lastUpdated, and the Activity Log to know what state a piece is actually in.

Gap 3: The "what was promised" surface doesn't exist

When I tell you "we'll 301 this piece" mid-conversation, that commitment vanishes when the session ends unless I either (a) immediately make the change or (b) write it to Notion. In the brand-overhaul case, I apparently committed verbally and then the session compacted/ended before I wrote the redirect. There's no "pending commitments" log anywhere, so when I resume work in a new session, I rebuild the to-do from scratch instead of from a durable record.

This is the deepest structural gap because it's the one that produces the worst class of error: actions I claimed to take but didn't. Everything else is just inefficiency.

4. Five proposals, ordered by leverage

DO NOW Proposal 1: Pending Commitments database in Notion

Problem solved: Gap 3 (verbal commitments vanish on session end).

Mechanic: A new Notion database called "Pending Commitments" with columns: Commitment, Promised By Session, Target URL, Action Type (rewrite / 301 / delete / other), Status (pending / done / cancelled), Created, Completed.

How it works: Whenever I say "I'll do X" but can't immediately do it, I write a row. /seo-brain loads pending rows at session start. Sessions can't end with pending rows that were touched in-session unless the user explicitly defers them.

Cost: ~2 hours setup. One new Notion DB, one tool added to /seo-brain Step 1.

Side effect: Forces me to be more disciplined about what I commit to mid-conversation. Verbal "we'll do that" becomes a tracked row.

DO NOW Proposal 2: Publication-aware audit logic in skills

Problem solved: Gap 2 (lastUpdated conflates new + refreshed + patched).

Mechanic: The canonical "what's stale" query becomes:

// pseudocode
stale_candidates = published_blog_posts
  .filter(d => published_date < N days ago)        // exclude freshly-published
  .filter(d => last_updated < M days ago)          // exclude recently-refreshed
  .filter(d => !redirected(d.slug, next_config))   // exclude already 301'd
  .filter(d => !in_activity_log_last_N_days(d.slug)) // belt-and-suspenders

This 4-stage filter is what scripts/audit-publication-dates-v2.mjs does today. The proposal: codify it as the default audit logic that /seo-brain and any "what should I refresh next" workflow uses. Update CLAUDE.md with the rule: "Refresh candidacy requires all four filters to pass."

Cost: ~1 hour. Update the /seo-brain skill markdown + one inline rule in CLAUDE.md.

DO SOON Proposal 3: Activity Log digest worker

Problem solved: Activity Log will exceed reasonable fetch size within 2-3 months.

Mechanic: A worker tool (the Notion workers infrastructure already exists per scripts/notion-workers/) that runs nightly and:

  1. Reads Activity Log rows from the last 30 days
  2. Aggregates by skill, by URL touched, by outcome
  3. Writes a single "Digest: YYYY-MM-DD" page that summarizes the period
  4. Optionally archives older Activity Log rows to a separate "Activity Log Archive" DB

What skills read: Instead of fetching all rows, fetch (a) the latest digest + (b) only the last 7 days of raw rows. Total context cost: ~5 KB instead of 50+ KB.

Cost: ~1 day. New worker tool + skill update + one-time backfill.

DO SOON Proposal 4: Per-skill context scoping

Problem solved: Right now /seo-brain fetches a fixed bundle of context regardless of what the user actually asked. Big and getting bigger.

Mechanic: Add a routing layer to /seo-brain that examines the user's intent and loads only the relevant subset:

User intentLoad only
"What should we write next?"Strategy page summary + content calendar (next 14 days) + last 7d Activity Log
"Audit what's stale"publishedDate + lastUpdated for all blog posts + redirect map. Nothing else.
"What's working?"GSC overview + Peec status + Targets + Daily Snapshots (last 14d only)
"Brainstorm content"Strategy page + Patterns Registry + KB insights (Active filter)
"Ship this draft"The single calendar entry + voice rules. Nothing else.

Cost: ~half day. Significant skill update. Pay off compounds as context budget pressures grow.

DO LATER Proposal 5: A "Session State" file at project root

Problem solved: Local session amnesia — when a new Claude session starts, it doesn't know what the previous session was working on.

Mechanic: A .claude/session-state.json file that gets updated by each skill as it works. Records: current sprint, last batch shipped, what's pending, what the next planned action is.

How a new session uses it: First action of a new session reads this file. Shows the user: "Last session ended after shipping batch 13. Next planned action was batch 14 with these 4 pieces. Continue?"

Cost: ~1 day, plus discipline to keep it updated. Higher ongoing maintenance cost than the Pending Commitments DB but covers more state.

Trade-off vs Proposal 1: Pending Commitments is durable but narrow (just commitments). Session State is broader but local-only. They're complementary, not alternatives. Start with Pending Commitments because it survives across machines.
