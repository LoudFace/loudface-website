Architecture Recommendation

Stay on Notion. Don't build Postgres. Here's why, and the exact path to "impeccable" without over-engineering.

The recommendation, in one paragraph. Don't build a Postgres database or a custom app. The problems we identified — Activity Log bloat, lost commitments, context window pressure, skill memory gaps — are shape problems, not storage problems. They get solved by adding a thin digest layer on top of the worker infrastructure that already exists in this repo (scripts/notion-workers/), a small new "Pending Commitments" Notion database, and skill-level discipline around publication-aware audits and context scoping. Total focused effort: ~3 days. Total new infrastructure to maintain: zero. If we hit specific scaling triggers in 12+ months (laid out below), revisit Postgres then. Don't pre-build for problems you don't have.

The trap with "build a real app" is that it sounds disciplined but is actually less disciplined. It commits us to UI work, hosting, backups, schema migrations, and a permissions layer to recover what Notion gives us for free. Worse: it removes you and the team from the loop because Notion's value isn't storage — it's that a human can open it, edit a row, and the system reflects the change immediately. Postgres would force you to either build a CRUD UI or route all edits through Claude. Both are downgrades.

1. The real question

The framing "Notion vs Postgres" implies one storage layer holds everything. That's not how the system actually works today, and it shouldn't be how it works tomorrow.

Right now we already use four storage layers in production:

LayerWhat it holdsWhy it's there
SanityPublished content (blog posts, case studies, team, clients)It's the CMS the live site reads from. Schema-validated, structured.
NotionStrategy, content calendar, KB insights, snapshots, patterns, targetsHuman-readable, human-editable, multi-user, no UI to build.
GSC + Peec AISearch performance + AI citation dataExternal analytics. We read; we don't write.

None of these are competing with each other. They're each optimized for what they hold. Postgres would replace Notion — but Notion isn't the part that's struggling. The part that's struggling is the shape of how Notion data flows into Claude sessions.

The honest reading: Notion loses on indexed queries at scale and aggregations. Everywhere else it's competitive or better. For our actual workload — under 1,000 active rows per database, human-edited inputs, low-frequency reads from skills — the indexed-query weakness doesn't matter yet.

3. The proposed architecture in one picture

Three-layer model — already 80% built, just needs to be made explicit

Layer 1: Human-curated

Notion. Source of truth for things humans edit.

  • Strategy page
  • Thought Leadership KB
  • Patterns Registry
  • Targets (KPIs)
  • Pending Commitments (new)
  • Content calendar (Website Content DB — metadata only)
Layer 2: Machine-generated, time-series

Notion. But ring-buffered + digested by workers.

  • Activity Log (last 7d raw + nightly digest)
  • Daily Snapshots (30d rolling + monthly digest)
  • Cusp Pages (current state only, not history)
  • Prompt Coverage (current state only)
  • Competitor Landscape (current state only)
Layer 3: Skill execution

Local + git. Per-session memory.

  • Skills with CHANGELOG sections
  • Per-skill context scoping (router pattern)
  • Optional .claude/session-state.json
  • Skill-level publication-aware audit logic

The key shift: Layer 2 currently treats every row as permanent. The proposal is to treat it as a ring buffer with automatic digesting. The Notion data stays — we just summarize old rows nightly so skills can read a digest instead of the raw stream.

How a session actually flows under this architecture

User: "what should I refresh next?"
  ↓
/seo-brain skill loads (routed to "audit" context, not full brain):
  - publishedDate + lastUpdated for all blog posts  [Sanity, fast]
  - redirects from next.config.ts                   [local file]
  - Pending Commitments rows (status=pending)       [Notion, ~10 rows]
  - Activity Log last-7d-digest page                [Notion, 1 page]
  ↓
Skill applies the 4-stage filter:
  - publishedDate > 30d ago
  - lastUpdated > 30d ago
  - slug not in redirects
  - slug not in last-7d Activity Log
  ↓
Returns: list of genuine candidates, plus any pending commitments
  that haven't been executed.
  ↓
Total tokens loaded: ~3-5KB. Today: ~30-50KB.

That's the entire architectural improvement. No new database engine. No new app. No new hosting. The "digest" pages are just regular Notion pages written by a worker on a cron.

4. The 5 fixes from yesterday, mapped to Notion-native execution

Fix 1: Pending Commitments DB

NotionSkill update

What: One new Notion database. Columns: Commitment, Target URL, Action Type (rewrite / 301 / delete / verify), Status (pending / done / cancelled), Created, Promised By Session.

How skills use it: /seo-brain reads Status = pending rows at startup and surfaces them. Skills can write new rows. Session can't cleanly end with pending rows from this session unless explicitly deferred.

Effort: 30 min to create the DB. 1 hour to update /seo-brain. Reads ~5-20 rows per session, trivial cost.

Fix 2: Publication-aware audit logic

Skill update

What: Update /seo-brain and any skill that does refresh-candidacy work. Hardcode the 4-stage filter. Update CLAUDE.md rule: "Refresh candidacy claims require publishedDate + lastUpdated + redirect-map + Activity-Log check."

Effort: 1 hour. Pure markdown + small skill logic update. scripts/audit-publication-dates-v2.mjs already exists as the reference implementation.

Fix 3: Activity Log digest worker

WorkerNotion

What: New tool inside scripts/notion-workers/. Runs nightly. Reads Activity Log rows from last 7 days. Aggregates by skill, by URL, by outcome. Writes a "Digest: YYYY-MM-DD" Notion page. Optionally archives rows older than 60 days to an "Activity Log Archive" DB.

How skills use it: /seo-brain reads the latest 1-2 digest pages instead of querying Activity Log directly. Total context cost: ~3KB vs ~50KB at 6-month scale.

Effort: ~1 day. The worker pattern (refreshCalendarMetrics, dailySnapshotsSync) is already established. This is one more tool in the same shape.

Fix 4: Per-skill context scoping (router pattern)

Skill update

What: /seo-brain currently loads a fixed bundle every time. The proposal: examine the user's actual question first, then load only the relevant subset.

How: The skill markdown starts with an intent-detection step. Five canonical intents map to five different fetch bundles. (Diagram in yesterday's investigation doc, Proposal 4.)

Effort: ~half day. Significant skill rewrite, but contained to one file.

Fix 5: Session State file (and skill CHANGELOG)

Local fileSkill convention

What: Two small additions. (a) .claude/session-state.json at repo root — each skill writes its last action. New sessions read it first. (b) Each skill's SKILL.md grows a ## Changelog section that humans (or Claude) update when behavior changes. /seo-brain reads its own changelog at load and shows last 2-3 entries.

Effort: Session state: ~half day. Skill changelog convention: trivial (just update the markdowns once and follow it).

6. The triggers that would make me change this recommendation

I want to be honest about when Postgres would actually be the right call. Three triggers, any one of which should make us revisit:

Trigger 1: Notion DB hits 5,000+ active rows. Today none of our DBs are close to this. Activity Log is the fastest-growing at ~10-20 rows/day, but the digest+archive pattern keeps active rows under 200 indefinitely. If we ever scale to managing 20+ client engagements in parallel and Activity Log balloons past digesting capacity, that's the signal.
Trigger 2: We need real-time aggregations across multiple databases. Example: "for every blog post, show me Peec citation count, GSC clicks, branded search lift on NEW queries, and which Patterns it tags into — sorted by combined score, refreshing every 5 minutes." Notion can't do this cleanly. Today our analysis runs in nightly batches; if we ever need live dashboards, that's the signal.
Trigger 3: We start selling AEO/SEO tooling externally. If LoudFace ever ships a product that lets other agencies run AEO programs (a "Peec for B2B SaaS" style SaaS), then yes — that needs a real database. But that's a different company than the one we're building right now. Don't pre-build for it.

None of these triggers are imminent. The 3-day Notion-native fix has 12+ months of runway. By the time any trigger fires, we'll have months of operating experience to inform what the Postgres schema should actually look like — instead of guessing now.

The honest summary

The temptation to build "the proper system" is real. It feels disciplined. But the system we have is already three coordinated layers (Sanity + Notion + workers + skills), and the problems we hit today aren't because Notion is wrong — they're because we didn't add a digest layer and a commitments surface that Notion was always going to need at this scale.

Build the digest. Add the commitments DB. Update the skills. Stop. If the triggers fire in 12 months, we revisit. If they don't, you saved a quarter of engineering time and the system runs impeccably because each layer does what it's actually good at.
